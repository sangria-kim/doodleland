# 그림놀이터 구현 현황 및 잔여 과제 (2026-03-30 기준)

## 0) 현재 상태 요약
- 홈, 캡처, 크롭, 저장, 라이브러리, 무대 진입까지 MVP 기본 흐름은 구현되어 있습니다.
- 규칙 기반 배경 제거, 썸네일 생성, Drift 저장, 무대 모션 재생, 드래그/탭/제거 상호작용도 동작합니다.
- 다만 문서와 달리 저장 전 화면은 투명 결과 미리보기가 아니라 크롭 결과 확인 단계입니다.

## 1) 구현 완료 범위

### 1-1. 공통 인프라
- Riverpod `ProviderScope`
- GoRouter 기반 기본 라우팅
- Drift `characters` 테이블 + DAO
- 캐릭터 파일 저장 경로 팩토리
- 카메라/갤러리 권한 매핑
- Android immersive fullscreen 공통 적용

### 1-2. 캡처 및 저장
- 카메라/갤러리 선택
- 네이티브 크롭 연동
- 규칙 기반 배경 제거
- 원본/투명 PNG/썸네일 저장
- 저장 후 홈 또는 재촬영 플로우

### 1-3. 라이브러리
- 캐릭터 조회
- 빈 상태 UI
- 길게 누르기 삭제
- 무대 연동용 캐릭터 선택 데이터 소스

### 1-4. 무대
- 배경 5종 선택
- 캐릭터 추가 바텀시트
- 모션 5종 선택
- 등장 애니메이션
- 탭 반응, 드래그 이동, 길게 눌러 제거
- 최대 10개 배치 제한

### 1-5. 테스트 / 기본 검증
- 배경 제거 테스트
- 저장 / 삭제 유스케이스 테스트
- 라이브러리 / 홈 / 배경 선택 / 스테이지 상태 테스트
- 릴리즈 체크리스트 문서 존재

## 2) 현재 기준 잔여 과제

### 2-1. UX 정합화
- 저장 전 화면에서 실제 투명 결과를 보여주도록 플로우를 재정리할지 결정 필요
- 저장 완료 스낵바 중심 경고 전달을 화면 내 안내로 승격할지 검토 필요

### 2-2. 구조 정리
- placeholder 파일 `image_processor.dart`, `stage_painter.dart`, `background_selector.dart` 정리 필요
- stage 렌더링 구조를 계속 widget 기반으로 유지할지, 별도 렌더링 계층으로 정리할지 판단 필요

### 2-3. 기능 확장 후보
- 캐릭터 이름 수정
- 무대 씬 저장 / 불러오기
- 모션 체계 재설계
- 전체 UI 리디자인

## 3) 운영 메모
- 코드 작업은 `main`이 아닌 `feature/...` 브랜치에서 진행합니다.
- 문서 전용 수정은 `main`에서 허용되지만 커밋 메시지 컨벤션은 동일하게 유지합니다.
- `main`에 새 커밋이 추가되면 `docs/PROGRESS.md`를 함께 갱신합니다.
